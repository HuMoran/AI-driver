# v0310-governance-workflow.spec.md

## Meta

- Date: 2026-04-21
- Review Level: B

## Goal

Close the governance workflow hole that caused v0.3.8 to ship without its approved `R-009` constitution amendment, forcing a separate v0.3.9 docs release. Three tightly related gaps:

1. **`/ai-driver:review-pr` does not detect governance replies.** It ingests existing reviewer comments into the `Existing reviewer findings` table but does not treat `R-00N` approval / rejection text as a **decision**, so a hand-composed verdict can still claim "governance approval pending" when an `approve as-is` is already posted. Risk: humans (and the main agent) miss an existing approval and proceed as if the amendment is still open.
2. **`/ai-driver:merge-pr` does not check governance state before merging.** It merges a PR whose body proposes an `R-00N` amendment even when no approval comment exists yet, and it does not detect approvals that DO exist. Result: either (a) the amendment gets lost (approval not applied before merge — the v0.3.9 mishap), or (b) an unapproved amendment merges unilaterally (a worse future failure mode).
3. **The pattern of "amendment-commit lands on the same PR after approval" is not enforced by tooling.** v0.3.6 R-008 did it manually; v0.3.8 R-009 did NOT (the amendment became a separate post-hoc PR). Convention is only documented in prose (`SHOULD-003` in prior specs). This spec promotes the pattern to a contract checkable by `merge-pr`.

After v0.3.10, a feature PR that proposes an amendment will reliably ship with its amendment commit in the same release, or fail-closed with a clear message if approval is missing.

## User Scenarios

### Scenario 1: `review-pr` detects governance replies automatically (Priority: P0)

**As a** maintainer running `/ai-driver:review-pr <N>` on a PR that proposes `R-00N` in its body,
**I want** the command to scan issue-comments for `approve as-is` / `approve with edits: ...` / `reject: ...` replies authored by an allowed reviewer,
**so that** the verdict accurately names the governance state (`R-00N: approved by @<login> at <ISO>` / `R-00N: rejected by @<login>: <reason>` / `R-00N: pending` / `R-00N: contested — see thread`) without me re-reading the thread by hand.

**Acceptance Scenarios:**

1. **Given** a PR body proposing `R-00N: <text>` and an issue-comment by a repo owner/admin that matches `^approve as-is$` / `^approve with edits:.*` / `^reject:.*` (case-insensitive, whitespace-trimmed), **When** `/ai-driver:review-pr` runs, **Then** the final verdict section has a `### Governance decisions` block naming each detected approval/rejection with author + ISO timestamp + commit-anchor suggestion (e.g., `land as docs(constitution): add R-00N`).
2. **Given** a PR body proposing `R-00N` with **no** matching approval comment, **When** review-pr runs, **Then** the verdict's Governance decisions block lists `R-00N: pending — no approval/rejection comment detected`.
3. **Given** conflicting replies on the same `R-00N` (an `approve as-is` and a later `reject: ...`), **When** review-pr runs, **Then** the block lists `R-00N: contested — see thread` and names both authors + timestamps. The command does NOT guess resolution; it surfaces the conflict for a human.

**Independent Test Method:** craft three fixture PRs (approved / pending / contested) and run review-pr; assert the verdict body contains the correct `R-00N:` line for each.

### Scenario 2: `merge-pr` blocks on unresolved governance proposals (Priority: P0)

**As a** maintainer running `/ai-driver:merge-pr <N>` on a PR whose body proposes `R-00N`,
**I want** the command to fail-closed unless **either** (a) a matching approval comment from a repo owner/admin exists AND the branch contains a commit amending `constitution.md` with `R-00N` in the diff, **or** (b) I pass `--accept-open-amendment <rationale>` to defer the amendment to a later PR,
**so that** v0.3.9-style accidents (approval silently missed, feature ships without amendment) cannot recur.

**Acceptance Scenarios:**

1. **Given** a PR proposing `R-00N` with an `approve as-is` comment and a commit `docs(constitution): add R-00N` on the branch, **When** merge-pr runs, **Then** it proceeds normally.
2. **Given** a PR proposing `R-00N` with an approval comment but **no** amendment commit on the branch, **When** merge-pr runs, **Then** it aborts with: `ERROR: R-00N is approved by @<login> but no amendment commit on this branch. Either add the commit (docs(constitution): add R-00N + template pair) or pass --accept-open-amendment "<reason>" to defer.`
3. **Given** a PR proposing `R-00N` with **no** approval comment, **When** merge-pr runs, **Then** it aborts with: `ERROR: R-00N is proposed in the PR body but no approval/rejection comment detected. Ask the maintainer to reply, or pass --accept-open-amendment "<reason>".`
4. **Given** `--accept-open-amendment "R-00N deferred to v<next>"` is passed, **When** merge-pr runs, **Then** the rationale is recorded in the merge commit message, release notes, and a new issue-comment on the PR so the deferral is audit-visible.

**Independent Test Method:** three fixture PR shapes (approved+commit, approved-no-commit, no-approval), run merge-pr in dry-run mode, assert the expected abort messages or proceed signal.

**Acceptance Scenarios:**

**As a** validator,
**I want** v0.3.10's own PR to include a proposed `R-010` (or to propose no new rule at all) **and** to go through merge-pr with the new governance check active,
**so that** the feature exercises itself on the first real PR.

**Acceptance Scenarios:**

### Edge Cases

- **Ambiguous approval author.** "approve as-is" posted by a bot or a drive-by contributor who is not a repo owner/admin. **Mitigation:** the approval-author allowlist is `gh api repos/<owner>/<repo>/collaborators` filtered to `permission in {admin, maintain}`. Other authors' approvals count as commentary, not approval.
- **Markdown rendering vs raw body.** GitHub stores comments as raw markdown; `gh api /issues/<n>/comments --jq .[].body` returns the raw text. Grep uses raw text, so code-block quoted `approve as-is` inside a comment counts as approval (by design — if you quoted it, you probably meant it; otherwise the maintainer can reject explicitly).
- **Multiple `R-00N` proposals in one PR body.** Rare but possible. Each proposal is detected independently (`^\s*####?\s+R-[0-9]{3,}:|^\s*\*\*R-[0-9]{3,}:`). Each needs its own approval + its own amendment commit.
- **Amendment commit references a different `R-XXX` than the PR body proposes.** Typo guard: the `merge-pr` check requires the SAME number, not just any `docs(constitution): add R-*`.

## Constraints

### MUST

- MUST-001: `review-pr` reads governance replies from `$STAGE/issue-comments.json` (staged in Gate 3), not from a fresh `gh api` call. This keeps the v0.3.8 stage-then-read invariant intact — all untrusted data passes through `$STAGE`.
- MUST-002: `merge-pr`'s governance check runs in Step 0 preflight (pure reads, no mutation). If it aborts, **nothing has been written** — same fail-closed shape as every other Step 0 abort.
- MUST-003: The approval-author allowlist is computed at runtime from `gh api repos/<owner>/<repo>/collaborators` filtered to `permission in {admin, maintain}`. It is NOT hardcoded. Non-admin approvals count as commentary, not governance decisions.
- MUST-004: When `--accept-open-amendment <rationale>` is used, the rationale is recorded in **three places**: merge commit message, release notes (via CHANGELOG if applicable), and a new PR comment posted before the merge. No single-location audit record.
- MUST-005: Amendment commit detection is by **exact R-number match**. `docs(constitution): add R-010` does NOT satisfy a PR body proposing `R-011`.

### MUST NOT

- MUSTNOT-001: Do NOT auto-apply the amendment on detection. `merge-pr` never writes to `constitution.md` — that is still a human-reviewed commit on the branch. The new check only verifies the commit exists and has the right R-number.
- MUSTNOT-002: Do NOT treat non-admin `approve as-is` as approval. Drive-by approvals are commentary.
- MUSTNOT-003: Do NOT weaken any v0.3.4 / v0.3.5 / v0.3.6 / v0.3.7 / v0.3.8 / v0.3.9 guardrail. Existing Trust boundary, path gates, subagent allowlists, stage-then-read, return-channel sanitization, R-008, R-009 all remain.
- MUSTNOT-004: Do not amend `constitution.md` in this spec unless a new operational rule is proposed in the PR body and approved. If no new rule is needed, `constitution.md` stays untouched; this feature is enforcement of existing governance, not a new rule.

### SHOULD

- SHOULD-001: The governance detection regex is tolerant of leading/trailing whitespace, case, and Markdown emphasis (`*approve as-is*` still counts).
- SHOULD-002: Conflict resolution (`contested` state) emits a structured finding in the PR review body with author + timestamp so humans can pick the resolution.

## Implementation Guide

Behavioural invariants only; exact grep / jq / shell details live in the command docs.

### `review-pr` — Governance decisions section

After Step 5 synthesis builds the final finding set, a new sub-step parses `$STAGE/issue-comments.json` (already staged in Gate 3) to extract governance decisions. For each `R-[0-9]+` mentioned in the PR body, walk all comments; a comment's body is a governance reply iff:

1. Its first non-blank, non-emphasis-stripped line matches `^(approve as-is|approve with edits:.*|reject:.*)$` (case-insensitive).
2. Its `user.login` is in the admins/maintainers allowlist (fetched once per run).

Multiple matching comments on the same R-number: resolve as `contested` unless all replies agree. Emit in the review comment body under `### Governance decisions` with lines like:

```
- R-009: approved by @HuMoran at 2026-04-20T11:12:13Z
  → Amendment commit needed: docs(constitution): add R-009
- R-010: pending — no approval/rejection comment detected
- R-011: contested — @HuMoran approved 2026-04-21T09:00Z; @other rejected 2026-04-21T10:00Z
```

### `merge-pr` — Step 0.10 Governance check

New preflight sub-step between existing 0.9 (next version) and the dry-run guard. Pseudocode:

```
for each R-NNN proposed in the PR body:
  detected_state = classify_replies(R-NNN, issue_comments, admin_allowlist)
  amendment_commit = git log main..HEAD -- constitution.md --format=%s \
                     | grep -Fq "docs(constitution): add R-NNN"
  case detected_state:
    approved:
      if amendment_commit: continue
      else if --accept-open-amendment <rationale>:
        record rationale in merge commit, release notes, PR comment
        continue
      else abort "R-NNN approved by @X but no amendment commit on this branch"
    pending:
      if --accept-open-amendment <rationale>:
        record, continue
      else abort "R-NNN proposed but no approval/rejection comment detected"
    contested:
      abort "R-NNN contested — see thread; resolve before merge"
```

All three abort cases print recovery hints — either "add the commit" or "pass the flag with rationale".

### Amendment commit helper

Document a one-shot helper in `AGENTS.md`'s governance section:

```bash
# After approval arrives on the PR:
RN=R-010  # the rule number
COMMIT_MSG="docs(constitution): add $RN — approved by @<login> in PR #<n>"
# Edit constitution.md + plugins/ai-driver/templates/constitution.md
# Then:
git add constitution.md plugins/ai-driver/templates/constitution.md
git commit -m "$COMMIT_MSG"
```

No script file is required — the shape is simple enough that a documented template suffices.
