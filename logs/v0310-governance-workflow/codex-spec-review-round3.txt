# v0310-governance-workflow.spec.md

## Meta

- Date: 2026-04-21
- Review Level: B

## Goal

Close the governance workflow hole that caused v0.3.8 to ship without its approved `R-009` constitution amendment, forcing a separate v0.3.9 docs release. Three tightly related gaps:

1. **`/ai-driver:review-pr` does not detect governance replies.** It ingests existing reviewer comments into the `Existing reviewer findings` table but does not treat `R-00N` approval / rejection text as a **decision**, so a hand-composed verdict can still claim "governance approval pending" when an `approve as-is` is already posted. Risk: humans (and the main agent) miss an existing approval and proceed as if the amendment is still open.
2. **`/ai-driver:merge-pr` does not check governance state before merging.** It merges a PR whose body proposes an `R-00N` amendment even when no approval comment exists yet, and it does not detect approvals that DO exist. Result: either (a) the amendment gets lost (approval not applied before merge — the v0.3.9 mishap), or (b) an unapproved amendment merges unilaterally (a worse future failure mode).
3. **The pattern of "amendment-commit lands on the same PR after approval" is not enforced by tooling.** v0.3.6 R-008 did it manually; v0.3.8 R-009 did NOT (the amendment became a separate post-hoc PR). Convention is only documented in prose (`SHOULD-003` in prior specs). This spec promotes the pattern to a contract checkable by `merge-pr`.

After v0.3.10, a feature PR that proposes an amendment will reliably ship with its amendment commit in the same release, or fail-closed with a clear message if approval is missing.

## User Scenarios

### Scenario 1: `review-pr` detects governance replies automatically (Priority: P0)

**As a** maintainer running `/ai-driver:review-pr <N>` on a PR that proposes `R-NNN` in its body,
**I want** the command to scan issue-comments for rule-scoped replies (`approve R-NNN as-is` / `approve R-NNN with edits: ...` / `reject R-NNN: ...`) authored by an allowed reviewer,
**so that** the verdict accurately names the governance state per-rule (`R-NNN: approved by @<login> at <ISO>` / `R-NNN: rejected by @<login>: <reason>` / `R-NNN: pending (<reason>)` / `R-NNN: contested — see thread`) without me re-reading the thread by hand.

**Acceptance Scenarios:**

1. **Given** a PR body proposing `R-NNN: <text>` and an issue-comment by a repo admin/maintainer whose **first substantive line** (after stripping fenced code and `> ` blockquote markers) matches `^approve R-NNN as-is$` (case-insensitive, whitespace-trimmed) AND whose `created_at` ≥ the PR's `updated_at`, **When** `/ai-driver:review-pr` runs, **Then** the final verdict has a `### Governance decisions` block listing `R-NNN: approved by @<login> at <ISO>` plus a commit-anchor suggestion (e.g., `land as docs(constitution): add R-NNN — approved by @<login> in PR #<n>`).
2. **Given** a PR body proposing `R-NNN` with **no** rule-scoped approval/rejection comment, **When** review-pr runs, **Then** the block lists `R-NNN: pending — no rule-scoped approval/rejection comment detected`.

### Scenario 2: `merge-pr` blocks on unresolved governance proposals (Priority: P0)

**As a** maintainer running `/ai-driver:merge-pr <N>` on a PR whose body proposes `R-NNN`,
**I want** the command to fail-closed unless the governance state is unambiguous,
**so that** v0.3.9-style accidents cannot recur AND the escape hatch does NOT recreate the original bug. **Three cases, explicit per case:**

- **Case A (approved + commit):** rule-scoped fresh approval comment exists AND the branch has `docs(constitution): add R-NNN …` commit whose diff adds the exact `R-NNN` text. **Proceed.**
- **Case B (approved, not yet landed):** approval exists but no amendment commit. **Allow `--defer-approved-amendment "<rationale>"`** (new flag, replaces the ambiguous `--accept-open-amendment`). Records rationale in three audit sinks; requires the maintainer to follow up with a constitution-only PR, as v0.3.9 did.
- **Case C (proposal open, no approval):** proposal text is in the PR body but no approval or rejection comment is present. **No flag allows merge.** The maintainer MUST either (c1) reach approval + land the amendment commit (→ Case A), or (c2) **remove the proposal block from the PR body** (so the PR no longer proposes the rule). merge-pr detects removal by re-reading the staged PR body after the maintainer edits and finding no `R-NNN:` proposal. This is what prevents the v0.3.9 failure mode — "shipped with a proposal still open" is never a valid state.
- **Case D (contested):** approval AND rejection both present. No flag allows merge. Maintainer must resolve in the thread first.

**Acceptance Scenarios:**

1. **Case A — Given** a PR proposing `R-NNN` with a rule-scoped fresh `approve R-NNN as-is` comment AND a `docs(constitution): add R-NNN …` commit whose `git show` diff contains the exact `R-NNN:` header, **When** merge-pr runs, **Then** it proceeds normally.
2. **Case B — Given** a PR proposing `R-NNN` with a fresh approval but **no** amendment commit, **When** merge-pr runs, **Then** it aborts with: `ERROR: R-NNN approved by @<login> at <ISO> but no "docs(constitution): add R-NNN …" commit on this branch. 3. **Case C — Given** a PR proposing `R-NNN` with **no** approval/rejection comment, **When** merge-pr runs, **Then** it aborts with: `ERROR: R-NNN proposed in PR body but no rule-scoped approval/rejection detected. No --defer flag applies here — either obtain approval + land the amendment commit, or remove the R-NNN block from the PR body. Ship-with-open-proposal is never allowed.`
4. **Case D — Given** a PR proposing `R-NNN` with both approval and rejection present, **When** merge-pr runs, **Then** it aborts with: `ERROR: R-NNN contested — approved by @A at <T1>, rejected by @B at <T2>. Resolve in the thread before merge.`

### Edge Cases

- **Ambiguous approval author.** `approve R-NNN as-is` posted by a bot or drive-by contributor who is not a repo owner/admin. **Mitigation:** the approval-author allowlist is `gh api repos/<owner>/<repo>/collaborators` filtered to `permission in {admin, maintain}`. Other authors' approvals count as commentary, not approval.
- **Markdown quoting / code fence false positives.** Examples of approval syntax inside a blockquote (`> approve R-009 as-is`) or a fenced code block (`` ``` approve R-009 as-is ``` ``) must NOT count as real approval. **Mitigation:** before matching, **delete** (not normalize) any line that begins with `> ` (blockquoted — removed entirely from the candidate stream) AND delete any content inside triple-backtick or triple-tilde fenced blocks. The approval marker must then appear as the **first substantive (non-empty, non-emphasis-only) line** of what remains. This is the "delete, don't strip" rule: a quoted line is erased rather than having its `> ` marker stripped, so a `> approve R-NNN as-is` never becomes matchable text.
- **Multiple `R-00N` proposals in one PR body.** Rare but possible. Each proposal is detected independently (`^\s*####?\s+R-[0-9]{3,}:|^\s*\*\*R-[0-9]{3,}:`). **Approval replies MUST name the specific rule number** — syntax is `approve R-NNN as-is` / `approve R-NNN with edits: ...` / `reject R-NNN: ...`. A bare `approve as-is` satisfies NO proposal (it's logged as commentary). Each R-NNN needs its own approval + its own amendment commit.
- **Amendment commit references a different `R-XXX` than the PR body proposes.** Typo guard: the `merge-pr` check requires the SAME number, not just any `docs(constitution): add R-*`.
- **Proposal text edited after approval.** The PR author edits the R-NNN block in the PR body AFTER an approval comment was posted. **Mitigation:** freshness is computed **per R-NNN proposal block digest**, NOT via the PR's generic `updated_at` (which changes for unrelated events like sync-with-main or metadata updates). For each proposed rule: (a) extract the R-NNN block text verbatim from the current PR body, (b) compute `DIGEST = sha256(block_text)`, (c) an approval comment is fresh iff its body contains `digest: <DIGEST>` (the reviewer includes the digest they approved — either copied from a review-pr suggestion or from manual verification). If the digest is absent from the approval, the approval is **always** treated as stale; review-pr emits an explicit prompt for the reviewer to re-approve with the current digest. Tradeoff: slightly heavier reviewer ergonomics in exchange for ignoring unrelated PR-update noise.
- **`approve R-NNN with edits: ...` interpretation.** This is NOT a plain approval; it's a **request-changes-then-approve**. Treated as `pending (edits requested by @X: <edit summary>)`. A later plain `approve R-NNN as-is` from the same or another admin is required to move the rule to `approved`. merge-pr aborts as `pending` otherwise.
- **Rationale-injection into commit/release notes.** `--accept-open-amendment "<rationale>"` writes caller-supplied text into merge-commit, release-notes, and a PR comment. **Mitigation:** cap `<rationale>` at 200 characters, reject newlines (must be a single line), and escape `` ` ``, `|`, `>`, `<`, `$` before interpolation. Longer rationales go into a PR comment only, never the commit or release-notes sink.

Behavioural invariants only; exact grep / jq / shell details live in the command docs.

### `review-pr` — Governance decisions section

After Step 5 synthesis builds the final finding set, a new sub-step parses `$STAGE/issue-comments.json` (already staged in Gate 3) to extract governance decisions. For each `R-[0-9]+` mentioned in the PR body, walk all comments; a comment's body is a governance reply iff:

1. Its first non-blank, non-emphasis-stripped line matches `^(approve as-is|approve with edits:.*|reject:.*)$` (case-insensitive).
2. Its `user.login` is in the admins/maintainers allowlist (fetched once per run).

Multiple matching comments on the same R-number: resolve as `contested` unless all replies agree. Emit in the review comment body under `### Governance decisions` with lines like:

```
- R-009: approved by @HuMoran at 2026-04-20T11:12:13Z
  → Amendment commit needed: docs(constitution): add R-009
- R-010: pending — no approval/rejection comment detected
- R-011: contested — @HuMoran approved 2026-04-21T09:00Z; @other rejected 2026-04-21T10:00Z
```

### `merge-pr` — Step 0.10 Governance check

New preflight sub-step between existing 0.9 (next version) and the dry-run guard. Pseudocode:

```
for each R-NNN proposed in the PR body:
  detected_state = classify_replies(R-NNN, issue_comments, admin_allowlist)
  amendment_commit = git log main..HEAD -- constitution.md --format=%s \
                     | grep -Fq "docs(constitution): add R-NNN"
  case detected_state:
    approved:
      if amendment_commit: continue
      else if --accept-open-amendment <rationale>:
        record rationale in merge commit, release notes, PR comment
        continue
      else abort "R-NNN approved by @X but no amendment commit on this branch"
    pending:
      if --accept-open-amendment <rationale>:
        record, continue
      else abort "R-NNN proposed but no approval/rejection comment detected"
    contested:
      abort "R-NNN contested — see thread; resolve before merge"
```

All three abort cases print recovery hints — either "add the commit" or "pass the flag with rationale".

### Amendment commit helper

Document a one-shot helper in `AGENTS.md`'s governance section:

```bash
# After approval arrives on the PR:
RN=R-010  # the rule number
COMMIT_MSG="docs(constitution): add $RN — approved by @<login> in PR #<n>"
# Edit constitution.md + plugins/ai-driver/templates/constitution.md
# Then:
git add constitution.md plugins/ai-driver/templates/constitution.md
git commit -m "$COMMIT_MSG"
```

No script file is required — the shape is simple enough that a documented template suffices.
