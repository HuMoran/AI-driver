# v0310-governance-workflow.spec.md

## Meta

- Date: 2026-04-21
- Review Level: B

## Goal

Close the governance workflow hole that caused v0.3.8 to ship without its approved `R-009` constitution amendment, forcing a separate v0.3.9 docs release. Three tightly related gaps:

1. **`/ai-driver:review-pr` does not detect governance replies.** It ingests existing reviewer comments into the `Existing reviewer findings` table but does not treat `R-00N` approval / rejection text as a **decision**, so a hand-composed verdict can still claim "governance approval pending" when an `approve as-is` is already posted. Risk: humans (and the main agent) miss an existing approval and proceed as if the amendment is still open.
2. **`/ai-driver:merge-pr` does not check governance state before merging.** It merges a PR whose body proposes an `R-00N` amendment even when no approval comment exists yet, and it does not detect approvals that DO exist. Result: either (a) the amendment gets lost (approval not applied before merge — the v0.3.9 mishap), or (b) an unapproved amendment merges unilaterally (a worse future failure mode).
3. **The pattern of "amendment-commit lands on the same PR after approval" is not enforced by tooling.** v0.3.6 R-008 did it manually; v0.3.8 R-009 did NOT (the amendment became a separate post-hoc PR). Convention is only documented in prose (`SHOULD-003` in prior specs). This spec promotes the pattern to a contract checkable by `merge-pr`.

After v0.3.10, a feature PR that proposes an amendment will reliably ship in one of three explicit states: **(A)** amendment commit landed in the same release, **(B)** maintainer explicitly defers the approved amendment via `--defer-approved-amendment` with a three-sink audit trail (matches the v0.3.9 shape but makes the deferral visible and intentional), or **(C)** the proposal block is removed from the PR body before merge. "Shipped with an unresolved open proposal" is never a valid state and is fail-closed.

## User Scenarios

**As a** maintainer running `/ai-driver:review-pr <N>` on a PR that proposes `R-NNN` in its body,
**I want** the command to scan issue-comments for rule-scoped replies (`approve R-NNN as-is` / `approve R-NNN with edits: ...` / `reject R-NNN: ...`) authored by an allowed reviewer (GitHub permission `admin` or `maintain`),
**so that** the verdict accurately names the governance state per-rule (`R-NNN: approved by @<login> at <ISO>` / `R-NNN: rejected by @<login>: <reason>` / `R-NNN: pending (<reason>)` / `R-NNN: contested — see thread`) without me re-reading the thread by hand.

**Acceptance Scenarios:**

1. **Given** a PR body proposing `R-NNN: <text>` and an issue-comment by a repo collaborator with permission `admin` or `maintain` whose body — **after deleting** any fenced code block and any line matching `^\s*>` (blockquote marker, with or without a trailing space; delete, not strip) — has `^approve R-NNN as-is digest: <sha256hex>$` (case-insensitive, whitespace-trimmed) as its first substantive line AND the `<sha256hex>` matches `sha256(current_R-NNN_block_text)`, **When** `/ai-driver:review-pr` runs, **Then** the final verdict has a `### Governance decisions` block listing `R-NNN: approved by @<login> at <ISO>` plus a commit-anchor suggestion (e.g., `land as docs(constitution): add R-NNN — approved by @<login> in PR #<n>`).
2. **Given** a PR body proposing `R-NNN` with **no** rule-scoped approval/rejection comment, **When** review-pr runs, **Then** the block lists `R-NNN: pending — no rule-scoped approval/rejection comment detected`.
3. **Given** conflicting rule-scoped replies on the same `R-NNN` (an `approve R-NNN as-is` and a later `reject R-NNN: ...`), **When** review-pr runs, **Then** the block lists `R-NNN: contested — see thread` and names both authors + timestamps. The command does NOT guess resolution.
4. **Given** an approval comment whose `digest: <hex>` value does NOT equal `sha256(current_R-NNN_block_text)` (PR body was edited after the approval, changing the digest), **When** review-pr runs, **Then** the block lists `R-NNN: pending (stale approval — digest mismatch; re-approve with digest: <current_hex>)`.

6. **Given** a comment saying `approve R-NNN with edits: clarify paragraph 2`, **When** review-pr runs, **Then** the block lists `R-NNN: pending (edits requested by @<login>: clarify paragraph 2)` — NOT approved. A plain `approve R-NNN as-is` from any admin is required to move to `approved`.

Caller must follow up with a constitution-only PR (as v0.3.9 did). Deferral is **visible and intentional**, not silent.

- **Case D (contested):** both approval and rejection present. No flag. Must resolve in thread.
- **Case E (rejected only):** `reject R-NNN: <reason>` exists, no approval, no contested state. **No flag.** merge-pr aborts with `ERROR: R-NNN rejected by @<login>: <reason>. Remove the proposal block from the PR body to continue, or withdraw the PR.` Same remediation as Case C: strip the proposal from the body.

**Acceptance Scenarios:**

1. **Case A — Given** a PR proposing `R-NNN` with a rule-scoped fresh `approve R-NNN as-is` comment AND a `docs(constitution): add R-NNN …` commit whose `git show` diff contains the exact `R-NNN:` header, **When** merge-pr runs, **Then** it proceeds normally.
2. **Case B — Given** a PR proposing `R-NNN` with a fresh approval but **no** amendment commit, **When** merge-pr runs, **Then** it aborts with: `ERROR: R-NNN approved by @<login> at <ISO> but no "docs(constitution): add R-NNN …" commit on this branch. **so that** `merge-pr`'s AC can mechanically detect an amendment commit and future archaeologists can trace the approval chain without hunting through comments.

**Acceptance Scenarios:**

2. **Given** `merge-pr.md` after this spec lands, **When** I read the governance check step, **Then** it specifies the **single-commit-set** detection procedure: `git log --format='%H %s' main..HEAD -- constitution.md` enumerates `<sha> <subject>` pairs, the harness filters to SHAs whose subject matches `^docs\(constitution\): add R-NNN `, then for each such SHA runs `git show <sha> -- constitution.md | grep -Eq '^\+.*R-NNN:'`. ### Edge Cases

- **Ambiguous approval author.** `approve R-NNN as-is` posted by a bot or drive-by contributor whose GitHub permission on the repo is neither `admin` nor `maintain`. **Mitigation:** the approval-author allowlist is `gh api repos/<owner>/<repo>/collaborators` filtered to `permission in {admin, maintain}` (exactly these two values — no `write`, no `triage`, no `read`). Other authors' approvals count as commentary, not approval.
- **Markdown quoting / code fence false positives.** Examples of approval syntax inside a blockquote (`> approve R-009 as-is`, `>approve R-009 as-is`, `   > approve R-009 as-is`, `>> approve R-009 as-is`) or a fenced code block (`` ``` approve R-009 as-is ``` ``) must NOT count as real approval. **Mitigation:** before matching, **delete** (not normalize) any line matching the regex `^\s*>` (zero-or-more whitespace, one-or-more `>` characters — covers `>foo`, `> foo`, `   > foo`, and nested `>> foo`) AND delete any content inside triple-backtick or triple-tilde fenced blocks (including the fence delimiters themselves). Blockquotes never wrap continuation lines in the comment grammar we accept (each quoted line must start with its own `>`); a non-quoted line following a quoted line terminates the blockquote. The approval marker must then appear as the **first substantive (non-empty) line** of what remains. This is the "delete, don't strip" rule: a quoted line is erased rather than having its `>` marker stripped, so `> approve R-NNN as-is` and `>approve R-NNN as-is` never become matchable text.
- **Multiple `R-00N` proposals in one PR body.** Rare but possible. Each proposal is detected independently (`^\s*####?\s+R-[0-9]{3,}:|^\s*\*\*R-[0-9]{3,}:`). **Approval replies MUST name the specific rule number** — syntax is `approve R-NNN as-is` / `approve R-NNN with edits: ...` / `reject R-NNN: ...`. A bare `approve as-is` satisfies NO proposal (it's logged as commentary). Each R-NNN needs its own approval + its own amendment commit.
- **Amendment commit references a different `R-XXX` than the PR body proposes.** Typo guard: the `merge-pr` check requires the SAME number, not just any `docs(constitution): add R-*`.
- **Proposal text edited after approval.** The PR author edits the R-NNN block in the PR body AFTER an approval comment was posted. **Mitigation:** freshness is computed **per R-NNN proposal block digest**, NOT via the PR's generic `updated_at` (which changes for unrelated events like sync-with-main or metadata updates). For each proposed rule: (a) extract the R-NNN block text verbatim from the current PR body, (b) compute `DIGEST = sha256(block_text)`, (c) an approval comment is fresh iff its body contains `digest: <DIGEST>` (the reviewer includes the digest they approved — either copied from a review-pr suggestion or from manual verification). If the digest is absent from the approval, the approval is **always** treated as stale; review-pr emits an explicit prompt for the reviewer to re-approve with the current digest. Tradeoff: slightly heavier reviewer ergonomics in exchange for ignoring unrelated PR-update noise.
- **`approve R-NNN with edits: ...` interpretation.** This is NOT a plain approval; it's a **request-changes-then-approve**. Treated as `pending (edits requested by @X: <edit summary>)`. A later plain `approve R-NNN as-is` from the same or another admin is required to move the rule to `approved`. merge-pr aborts as `pending` otherwise.
- **Rationale-injection into commit/release notes.** `--defer-approved-amendment "<rationale>"` writes caller-supplied text into merge-commit, release-notes, and a PR comment. **Mitigation:** cap `<rationale>` at 200 characters, reject newlines (must be a single line), and escape `` ` ``, `|`, `>`, `<`, `$`, backslash before interpolation. A rationale that exceeds 200 chars or contains a newline is **rejected at Step 0 with an abort** — no fallback "just write to PR comment" path exists (that would create an asymmetry between sinks and a smuggling channel). The maintainer must shorten the rationale or split the explanation: a brief summary goes in the flag, detailed context goes in a separate human-written PR comment.

Actual audit-sink writes (for `--defer-approved-amendment`) happen in a new Step 2.5 **after** CHANGELOG rewrite and **before** `gh pr merge`; they are wrapped in the same fail-closed error handling as the CHANGELOG rewrite.
- MUST-003: The approval-author allowlist is computed at runtime from the GitHub REST endpoint [`GET /repos/{owner}/{repo}/collaborators`](https://docs.github.com/en/rest/collaborators/collaborators#list-repository-collaborators) with `affiliation=all` and pagination handled via `gh api --paginate`. For each collaborator object returned, the allowlist INCLUDES the `login` iff **either** `.role_name` equals `"admin"` or `"maintain"` **or** (`role_name` absent) `.permissions.admin == true` or `.permissions.maintain == true`. Exact command: `gh api --paginate '/repos/{owner}/{repo}/collaborators?affiliation=all' --jq '.[] | select((.role_name == "admin") or (.role_name == "maintain") or (.permissions.admin == true) or (.permissions.maintain == true)) | .login'`. The legacy `permission` string field (values `admin|write|read|none`) is NOT used because it collapses `maintain` → `write` and would miss the maintain tier. Approvals from `login`s outside this allowlist count as commentary, not governance decisions.
- MUST-004: Governance reply syntax is **rule-scoped**: approvals and rejections MUST name the specific `R-NNN`. Bare `approve as-is` / `reject: ...` (no rule number) is commentary, not a governance decision. This prevents one approval from satisfying multiple proposals in a multi-proposal PR.
- MUST-005: Governance replies are parsed from the **first substantive line** of the comment body after applying the canonical "delete, don't strip" normalization: **delete entirely** (a) every line matching regex `^\s*>` (zero-or-more whitespace, one-or-more `>` — covers `> foo`, `>foo`, indented blockquotes, and nested `>>` quoting), (b) every line inside a triple-backtick or triple-tilde fenced block (including the fence delimiters). The remaining lines form the candidate stream; the first non-empty line must exactly match `^approve R-NNN as-is digest: <sha256hex>$` / `^approve R-NNN with edits:.*` / `^reject R-NNN:.*` (case-insensitive, whitespace-trimmed; no leading/trailing Markdown emphasis markers — emphasis is NOT tolerated). Approval syntax inside quoted or fenced examples NEVER counts.
- MUST-006: `approve R-NNN with edits: <edit summary>` is classified as `pending (edits requested)`, NOT `approved`. A later plain `approve R-NNN as-is` from any admin is required to reach `approved`.
- MUST-007: When `--defer-approved-amendment <rationale>` is used, the rationale is sanitized: **≤ 200 characters, single line (no newlines), with `` ` ``, `|`, `>`, `<`, `$` escaped** before interpolation into any audit sink. Longer or multi-line rationales are rejected at Step 0 with an error.
- MUST-008: `--defer-approved-amendment` rationale is recorded in **three audit sinks**: merge commit message (as a `Governance-deferral: <rationale>` trailer), release-notes section (under a `### Governance deferrals` subheading), and a new PR comment. All three writes occur in Step 2.5.
- MUST-009: Amendment commit detection is **single-commit-set AND content-matching**: the subject match, diff-content match, AND digest match MUST all come from the same commit SHA. Procedure: (a) `git log --format='%H %s' main..HEAD -- constitution.md` gathers candidate `<sha> <subject>` pairs; (b) filter to subjects matching `^docs\(constitution\): add R-NNN ` for the exact `R-NNN`; (c) for each matching SHA, extract the added `R-NNN` block from `git show <sha> -- constitution.md` — the contiguous range of `^\+` lines starting at the first `^\+.*R-NNN:` line and ending at the next `^\+\s*$` blank-added line or the next added `^\+###` heading (whichever comes first), then strip the leading `+` from each line; (d) apply the same canonicalization rules as MUST-013 (see **Proposal-block canonical form** below) to the extracted content and compute `sha256`; (e) this computed digest MUST equal the digest recorded in the rule-scoped approval comment. The detector passes iff at least one SHA satisfies **all three**: subject match, diff-add `R-NNN:` present, AND computed-digest == approval-digest. A commit that adds a different rule body under the approved `R-NNN:` header fails the detector and merge-pr aborts with `ERROR: R-NNN amendment commit <sha> body does NOT match approved digest <approval-digest>; committed digest <computed-digest>. Rewrite the commit to match the approved text or re-request approval for the new text.`
- MUST-010: **`merge-pr` stage-then-read invariant.** Governance check data — PR body, rule-scoped reply comments — is read from staged files under a per-run `mktemp -d` tempdir, NOT from fresh `gh api` calls whose stdout lands in the main session's Bash tool output. Same shape as `/ai-driver:review-pr` Step 2 (v0.3.8): `gh ... > "$STAGE/meta.json" 2> "$STAGE/meta.json.err"` etc. This keeps the R-009 sandbox executor contract in force at the merge path.
- MUST-011: **Case C has no escape flag.** If a PR body proposes `R-NNN` and no approval/rejection exists, merge-pr aborts unconditionally. The maintainer must either obtain approval (→ Case A or B) or remove the proposal block from the PR body (detected by re-reading the staged `meta.json` after their edit). `--defer-approved-amendment` applies only to Case B.
- MUST-012: **Audit-sink write order is fixed and idempotent.** When `--defer-approved-amendment` is used: PR comment first (idempotent via `<!-- ai-driver-defer:R-NNN -->` marker), then CHANGELOG `### Governance deferrals` subheading (atomic with the release commit), then merge commit trailer (`Governance-deferral: R-NNN: <rationale>` via `gh pr merge --body`). Partial-failure recovery contract is documented per-sink.
- MUST-013: **Approval freshness uses proposal-block digest, not PR `updated_at`.** Approval comments must include `digest: <sha256(canonical_R-NNN_block_text)>` (lowercase hex, 64 chars) to count as fresh. `review-pr` emits a suggestion line showing the current digest so the maintainer can include it by copy-paste. This eliminates the false-staleness triggered by unrelated PR-metadata events. No `created_at`/`updated_at` timestamp comparison is used; digest match is the sole freshness test. **Proposal-block canonical form** (used for both digest computation and content-match verification in MUST-009):
    - **Boundary**: the block starts at the line matching `^\s*####?\s+R-NNN:|^\s*\*\*R-NNN:` (the proposal header) and ends at the **first** of: (a) the next Markdown heading line `^#{1,6}\s`, (b) a horizontal rule `^\s*---\s*$`, (c) end of PR body. Both boundary markers are EXCLUDED from the block (the block is the content between markers).
    - **Line endings**: normalize CRLF (`\r\n`) and CR (`\r`) to LF (`\n`) before hashing.
    - **Trailing whitespace**: strip trailing spaces and tabs from each line (`s/[ \t]+$//`).
    - **Leading/trailing blank lines**: remove empty lines (lines matching `^\s*$`) from the top and bottom of the block; interior blank lines are preserved.
    - **Final newline**: the canonical form ends with exactly one LF.
    - **Encoding**: UTF-8, NFC-normalized (Python `unicodedata.normalize('NFC', s)` equivalent) before hashing.
  These five rules are applied in the listed order; `review-pr` implements them verbatim.
- MUST-014: **Governance flow requires merge-commit strategy.** When any `R-NNN` proposal is detected in the PR body, `merge-pr` MUST use `gh pr merge --merge` (not `--squash`, not `--rebase`). If the caller passes `--squash` on a governance PR, Step 0 aborts with `ERROR: governance PRs require --merge (merge-commit strategy) because audit sinks depend on a merge-commit trailer. Remove --squash or resolve the R-NNN proposal before merging.` This precondition also applies to `--defer-approved-amendment`: the third audit sink (merge-commit trailer) is undefined under squash/rebase.
- MUST-015: **Echoed-comment sanitization.** Any substring copied from a staged issue-comment into `review-pr`'s `### Governance decisions` output or `merge-pr`'s abort messages — specifically `<reason>` from `reject R-NNN: <reason>` and `<edit summary>` from `approve R-NNN with edits: <edit summary>` — MUST be (a) truncated to 120 characters (anything past the 120th character is dropped, and a trailing `…` is appended if truncation occurred), (b) stripped of ANSI escape sequences (`s/\x1b\[[0-9;]*[a-zA-Z]//g`), (c) stripped of control characters (ASCII < 0x20 except `\t`) which are replaced with a single space, (d) have `` ` ``, `|`, `$`, `<`, `>`, backslash escaped (`` \`  \|  \$  \<  \>  \\ ``) before interpolation into any abort message that is later rendered in a shell, markdown comment body, or merge-commit trailer. Rationale: issue-comment bodies are attacker-controlled; echoing them raw into review output or abort text is an output-channel injection vector. Instead of raw bodies, `review-pr` MAY emit a short canonical form plus a URL to the original comment (`https://github.com/<owner>/<repo>/pull/<n>#issuecomment-<id>`) for humans who need to read the full reason.

- MUSTNOT-001: Do NOT auto-apply the amendment on detection. `merge-pr` never writes to `constitution.md` — that is still a human-reviewed commit on the branch. The new check only verifies the commit exists and has the right R-number.
- MUSTNOT-002: Do NOT treat non-admin `approve as-is` as approval. Drive-by approvals are commentary.
- MUSTNOT-003: Do NOT weaken any v0.3.4 / v0.3.5 / v0.3.6 / v0.3.7 / v0.3.8 / v0.3.9 guardrail. Existing Trust boundary, path gates, subagent allowlists, stage-then-read, return-channel sanitization, R-008, R-009 all remain.

### SHOULD

- SHOULD-001: The governance detection regex is tolerant of leading/trailing whitespace and case only. Markdown emphasis markers are NOT tolerated — a line like `*approve R-NNN as-is digest: <hex>*` does not match the required regex and counts as commentary. Rationale: MUST-005 requires exact regex match on the first substantive line; allowing emphasis would contradict it. Reviewers who want to emphasize their decision can use a separate sentence.

## Implementation Guide

Behavioural invariants only; exact grep / jq / shell details live in the command docs.

### `review-pr` — Governance decisions section

After Step 5 synthesis builds the final finding set, a new sub-step parses `$STAGE/issue-comments.json` (already staged in Gate 3) to extract governance decisions. For each `R-[0-9]+` mentioned in the PR body, walk all comments; a comment's body is a governance reply iff:

1. After applying the "delete, don't strip" normalization (delete lines matching `^\s*>` — covers `> foo`, `>foo`, indented and nested blockquotes — and delete content inside triple-backtick / triple-tilde fences), its first non-blank line matches exactly one of: `^approve R-NNN as-is digest: <sha256hex>$` / `^approve R-NNN with edits:.*` / `^reject R-NNN:.*` (case-insensitive, whitespace-trimmed, no Markdown emphasis tolerance) — where `R-NNN` is the specific rule number from the PR body.
2. Its `user.login` is in the `admin|maintain` allowlist — the exact set returned by `gh api repos/<owner>/<repo>/collaborators` filtered to `permission in {admin, maintain}`, fetched once per run.
3. For approvals: `<sha256hex>` matches `sha256(current_R-NNN_block_text)`. Mismatched digest → classified as `pending (stale approval — digest mismatch)`.

Multiple matching comments on the same R-number: resolve as `contested` unless all replies agree. Emit in the review comment body under `### Governance decisions` with lines like:

```
- R-009: approved by @HuMoran at 2026-04-20T11:12:13Z
  → Amendment commit needed: docs(constitution): add R-009
- R-010: pending — no approval/rejection comment detected
- R-011: contested — @HuMoran approved 2026-04-21T09:00Z; @other rejected 2026-04-21T10:00Z
```

### `merge-pr` — Step 0.10 Governance check

New preflight sub-step between existing 0.9 (next version) and the dry-run guard. Pseudocode (validation-only; writes happen in Step 2.5):

```
# Precondition: reject --squash / --rebase when any R-NNN proposal is detected.
if proposals_present and merge_strategy != "--merge":
  abort "governance PRs require --merge (merge-commit strategy); audit sink (iii) depends on a merge-commit trailer"

for each R-NNN proposed in the PR body:
  detected_state = classify_replies(R-NNN, issue_comments, admin_allowlist)
  # Single-commit-set detection — subject + diff must come from the same SHA.
  amendment_shas = git log --format='%H %s' main..HEAD -- constitution.md \
                   | awk '$2 ~ /^docs\(constitution\):$/ && $0 ~ "add R-NNN "{print $1}'
  amendment_commit = any(
    sha for sha in amendment_shas
    if git show sha -- constitution.md | grep -Eq '^\+.*R-NNN:'
  )
  case detected_state:
    approved (digest-fresh):
      if amendment_commit: continue
      else if --defer-approved-amendment <rationale>:
        defer to Step 2.5 for writes (this step is validation-only)
        continue
      else abort "R-NNN approved by @X but no amendment commit on this branch"
    pending — no-reply subcase (Case C):
      abort "ERROR: R-NNN proposed but no rule-scoped approval/rejection detected. No --defer flag applies — either obtain approval + land the amendment commit, or remove the R-NNN block from the PR body."
    pending — stale-digest subcase (approval exists but digest mismatch):
      abort "ERROR: R-NNN approval by @X is stale — approval digest <old> does not match current proposal digest <new>. PR body was edited after approval. Re-request approval with the current digest (shown in review-pr output) or revert the R-NNN block edit."
    pending — edits-requested subcase:
      abort "ERROR: R-NNN has only approve-with-edits feedback from @X: <edit summary>. This is not a plain approval — obtain a plain 'approve R-NNN as-is digest: <hex>' comment from an admin/maintain collaborator, or remove the R-NNN block from the PR body."
    contested:
      abort "R-NNN contested — see thread; resolve before merge"
    rejected:
      abort "R-NNN rejected by @X: <reason>. Remove the proposal block from the PR body to continue, or withdraw the PR."
```

Recovery hints: Case B prints "add the commit OR pass `--defer-approved-amendment`". Cases C / D / E never print a flag hint — the only remediations are "get approval + land commit" or "remove the proposal block from the PR body" (or withdraw in Case E).
